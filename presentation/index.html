Fases de Implementación

Fase 1: Etapa Piloto

Análisis Bajo Demanda

Ejecución: Docker local por analista

  • ✓ Deploy en máquina local del analista
  • ✓ Análisis bajo demanda por repositorio
  • ✓ Generación manual de reportes
  • ✓ Sin automatización CI/CD
  • ✓ Control total del analista
  • ✓ Pruebas de viabilidad

Duración: 1-3 meses
Objetivo: Validar metodología y resultados

Fase 2: Producción

Análisis Automatizado

Ejecución: Servidor en producción con pipeline CI/CD

  • ✓ Deploy en infraestructura dedicada
  • ✓ Análisis automático en cada push
  • ✓ Integración con GitHub Actions/GitLab CI
  • ✓ Reportes automáticos por email
  • ✓ Alertas en tiempo real
  • ✓ Dashboard centralizado 24/7
  • ✓ Escalabilidad horizontal

Duración: Implementación gradual
Objetivo: Automatización completa del pipeline

Transición de Fases

Comandos de Despliegue

-- Local Development / Fase 1
$ docker-compose up -d
# Servidor en http://localhost:3000
-- Production Build / Fase 2
$ docker build -t scr-agent/app:v1.0.0 .
$ docker push docker-registry/scr-agent:v1.0.0
-- CI/CD Integration (GitHub Actions)
name: SCR Agent Analysis
on: [push, pull_request]
jobs:
analyze:
runs-on: ubuntu-latest
steps:
- uses: actions/checkout@v3
- run: docker run scr-agent/app:latest /app/analyze.js
- run: curl -X POST ${{ secrets.WEBHOOK_URL }} -d @report.json
-- Kubernetes Deploy
$ kubectl apply -f deployment.yaml
$ kubectl scale deployment scr-agent --replicas=10

Ejemplos & Casos de Uso

Vulnerabilidades reales detectadas y remediadas

01

SQL Injection

Detectado en: auth/login.js:45

❌ Código Vulnerable: 
const query = `SELECT * FROM users WHERE email = '${email}'`;
✅ Solución: 
const query = 'SELECT * FROM users WHERE email = ?';
db.query(query, [email]);

Severidad: CRÍTICO

CVSS Score: 9.8 (Crítico)

02

XSS (Cross-Site Scripting)

Detectado en: components/UserProfile.jsx:78

❌ Código Vulnerable: 
<div dangerouslySetInnerHTML={{__html: userInput}} />
✅ Solución: 
<div>{userInput}</div>

Severidad: ALTO

CVSS Score: 6.1 (Medio)

03

Credenciales en .env

Detectado en: config/.env:12

❌ Código Vulnerable: 
DATABASE_URL=postgres://user:password123@db.com
✅ Solución: 
// .env.example (committed)
DATABASE_URL=postgres://user:PASSWORD@db.com

// Add to .gitignore
.env
.env.*.local

Severidad: ALTO

Impacto: Exposición de credenciales de BD
